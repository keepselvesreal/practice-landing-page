
# 문서 수정 작업 안내 가이드

## 1. 문서 개요

### 목적
AI와의 상호작용을 통해 프로젝트 문서를 체계적이고 실수 없이 수정하기 위한 가이드

### 적용 범위
- PRD (Product Requirements Document)
- 아키텍처 문서
- 기술 사양서
- 기타 모든 프로젝트 문서

### 핵심 원칙
- **AI 지원 협업**: AI가 영향 범위와 고려사항을 파악하여 제시, 사용자와 대화를 통해 결정
- **단계적 진행**: 상위/핵심 개념부터 하위/파생 내용으로 점진적 결정
- **명시적 합의**: 각 단계의 결정사항은 명확히 문서화하여 사용자 동의 획득
- **최소 수준 유지**: 기존 문서의 상세도, 스타일, 형식 준수
- **추적 가능성**: 모든 수정사항의 근거와 버전 이력 유지
- **무중단 진행**: 확인 필요한 경우 외 단계 간 중단 없이 자동 진행

---

## 2. 작업 프로세스

### Step 1: 수정 요청 접수 및 명확화

**AI 수행 작업:**

**1-1. 초기 진입 처리**
- 사용자가 수정사항을 밝히지 않은 경우:
  ```
  AI: "어떤 문서의 어떤 부분을 수정하시려고 하나요?"
  ```

**1-2. 수정 내용 명확화**
- 사용자 요청이 불명확한 경우: 명확화 질문 진행
- 사용자 요청이 명확한 경우: 바로 다음 단계로

**1-3. 이해 내용 확인**
사용자 요청을 검토 후 AI가 이해한 내용을 정리하여 제시:
```
AI: "이해한 내용을 확인하겠습니다:
- [AI가 이해한 수정 내용 요약]

맞나요?"
```

**예시:**
```
사용자: "주문 취소 기능을 추가해야 해"

AI: "이해한 내용을 확인하겠습니다:
- 주문 확인 이메일에 주문 조회 링크 추가
- 해당 페이지에서 주문 정보 확인 및 취소 기능 제공

맞나요?"
```

---

### Step 2: 영향 범위 자동 분석

**AI 수행 작업:**

**2-1. 문서 구조 파악**
- 현재 수정 대상 문서 전체 검토
- 문서의 섹션 구조 파악

**2-2. 영향받는 요소 식별**
- 관련 섹션 자동 식별
- 영향받을 수 있는 요구사항/제약사항 파악
- 수정으로 인한 파생 영향 분석

**2-3. 관련 문서 식별**
- docs 폴더 구조 탐색
- 연관된 다른 문서 식별 (아키텍처, 기술 사양 등)
- 각 문서의 수정 필요 여부 판단

**2-4. 분석 결과 제시**

```
📊 영향 범위 분석 결과

[현재 문서] 수정 필요 섹션:
- ✅ [섹션명]: [수정 사유]
- ✅ [섹션명]: [수정 사유]
- ⚠️ [섹션명]: [검토 필요 사유]

[관련 문서] 영향받는 문서:
- [문서명]: [수정 필요 내용]
- [문서명]: [검토 필요 내용]

고려사항:
1. [고려사항 1]
2. [고려사항 2]
3. [고려사항 3]
```

### Step 3: 작업 계획 수립

**AI 수행 작업:**

**3-1. 작업 단계 분할**
- 수정 작업을 논리적 단계로 분할
- 상위/핵심 개념 → 하위/파생 내용 순서로 배치

**3-2. 의사결정 항목 명시**
- 각 단계별로 사용자가 결정해야 할 항목 명시
- 필요한 정보나 선택지 제시

**3-3. 작업 계획 제시 및 첫 단계 즉시 시작**

```
📋 작업 계획

▶️ Step 1: [단계명]
   - [ ] [의사결정 항목 1]
   - [ ] [의사결정 항목 2]

⬜ Step 2: [단계명]
   - [ ] [의사결정 항목 1]

⬜ Step 3: [단계명]
   - [ ] [의사결정 항목 1]

---
Step 1 시작합니다.
[첫 단계 내용 즉시 진행]
```

**예시:**
```
📋 작업 계획

▶️ Step 1: 취소 정책 및 제약사항 정의
   - [ ] 취소 가능 기간 결정
   - [ ] 취소 불가 조건 정의
   - [ ] 환불 처리 시간 결정

⬜ Step 2: 기능 요구사항 작성
   - [ ] Must Have 요구사항 추가
   - [ ] 관련 기존 요구사항 수정

⬜ Step 3: 리스크 및 의존성 업데이트
   - [ ] 새로운 리스크 식별
   - [ ] 기술적 의존성 추가

---
Step 1: 취소 정책 및 제약사항 정의를 시작합니다.

기존 환불 정책:
"배송 중 문제로 상품 불량 시에만 환불 가능 
(3일 이내 요청 + 미사용 상태 + 상품 반송 후 환불 처리)"

주문 취소 기능을 위한 정책이 필요합니다:
1. 취소 가능 기간: 언제까지 취소를 허용할까요?
2. 취소 불가 조건: 어떤 경우 취소가 불가능한가요?
3. 환불 처리: 얼마나 빨리 환불 처리할까요?
```

### Step 4: 단계별 내용 결정

**AI 수행 작업:**

**4-1. 진행 상태 표시**
매 단계 시작 시 작업 계획 목차를 다시 제시하여 현재 위치 표시:
```
📋 작업 계획

✅ Step 1: 취소 정책 정의 (완료)
▶️ Step 2: 기능 요구사항 작성 (진행 중)
⬜ Step 3: 리스크 업데이트
```

**4-2. 기존 문서 수준/스타일 유지**
- 기존 문서의 상세도 분석
- 동일한 문체, 형식, 구조 적용
- 기존 용어 및 표현 방식 준수

예시:
- 기존 문서가 표를 사용하면 → 표 형식 사용
- 기존 문서가 불릿을 사용하면 → 불릿 형식 사용
- 기존 요구사항이 "REQ-F##" 형식이면 → 동일 형식 유지

**4-3. 결정사항 명확히 문서화**
각 단계에서 결정된 내용을 명확한 문구로 정리:
```
Step [N] 결정사항:

[결정 항목 1]
- 결정 내용: [구체적 내용]
- 근거: [이유]

[결정 항목 2]
- 결정 내용: [구체적 내용]
- 근거: [이유]

이 내용으로 진행할까요? 수정사항이 있으신가요?
```

---

### Step 5: 최종 수정사항 정식화

**AI 수행 작업:**

**5-1. 전체 결정사항 종합**
모든 단계에서 결정된 내용을 종합하여 정리

**5-2. 추가/수정/삭제 명확히 구분**
문서에 반영될 변경사항을 구분하여 제시:

```
✅ 최종 수정사항 확정

【추가】
[섹션명] > [하위섹션명]:
- [추가할 내용 1]
- [추가할 내용 2]

[섹션명]:
- [항목명]: [추가할 내용]

【수정】
[기존 내용]
→ [새 내용]

[항목명]: [기존 내용]
→ [항목명]: [새 내용]

【삭제】
[섹션명]:
- [삭제할 내용]
```

**5-3. 사용자 최종 동의**
```
이 내용으로 문서를 수정하시겠습니까?
```

---

### Step 6: 문서 파일 생성 및 수정

**AI 수행 작업:**

**6-1. 기존 파일 아카이브**
1. 기존 파일을 해당 문서 폴더의 `inbox/` 서브폴더로 이동
2. 파일명에 수정 날짜 추가 (YY-MM-DD 형식)

**파일명 규칙:**
```
원본 경로: docs/prd/cosmetics_landing_mvp.md
아카이브: docs/prd/inbox/cosmetics_landing_mvp--25-10-17.md
```

**6-2. 새 버전 파일 생성**
1. 원본과 동일한 파일명으로 새 파일 생성
2. 합의된 수정사항 반영
3. 수정된 부분에 표시 추가

**수정 부분 표시:**

**Markdown 문서:**
```markdown
✨(추가) **REQ-F16**: 고객은 주문을 취소할 수 있어야 함
```

**코드 파일:**
```python
# ✨(추가) 주문 취소 API 추가
def cancel_order(order_id):
    pass
```

```html
<!-- ✨(추가) 주문 취소 버튼 추가 -->
<button>주문 취소</button>
```

**원칙:**
- Markdown: 이모지를 내용 앞에 직접 표시
- 코드: 해당 언어의 주석 형식으로 표시

**6-3. 메타데이터 업데이트**

**version 처리:**
```yaml
# 기존 파일
version: 1

# 새 파일
version: 2
```

**created_at 처리:**
```yaml
# 새 파일
created_at: 2025-10-17 15:30:00  # 현재 날짜/시간
```

**status 처리:**
```yaml
# 아카이브된 이전 파일 (inbox/)
status: Superseded

# 새 파일
status: Active
```

**links 처리:**
```yaml
# 새 파일
links:
  in: [
    inbox/cosmetics_landing_mvp--25-10-17.md,  # 이전 버전 추가
    ../index.md,
  ]
  out: [
    # 기존 out 링크 유지
  ]
```

**notes 처리:**
```yaml
# 새 파일
notes: "주문 취소 기능 추가 (v1에서 수정)"
```

### Step 7: 관련 문서 업데이트

**AI 수행 작업:**

**7-1. 기존 파일의 메타데이터 업데이트**
아카이브된 이전 버전 파일의 메타데이터 수정:
```yaml
# inbox/cosmetics_landing_mvp--25-10-17.md
---
created_at: 2025-10-15 20:39:42
version: 1
status: Superseded  # Active → Superseded
address: "k-cosmetics_landing_page/prd"
links:
  in: [../index.md]
  out: [
    ../cosmetics_landing_mvp.md,  # 새 버전 추가
  ]
tags: [prd, mvp, landing-page, k-beauty, philippines]
notes: "필리핀 시장 대상 K-뷰티 화장품 랜딩 페이지 MVP"
---
```

**7-2. index.md 업데이트**
프로젝트 인덱스 파일 수정:

```markdown
### 📋 기획 문서
- [PRD: 화장품 랜딩페이지 MVP](./prd/cosmetics_landing_mvp.md) (v2)

### 🏗️ 아키텍처 문서
- [시스템 아키텍처](./architecture/system_architecture.md) (v1)

### 📝 변경 이력

**PRD: 화장품 랜딩페이지 MVP**
- [주문 취소 기능 추가 (2025-10-17)](./prd/inbox/cosmetics_landing_mvp--25-10-17.md)

**시스템 아키텍처**
- [주문 취소 플로우 추가 
(2025-10-17)](./architecture/inbox/system_architecture--25-10-17.md)
```

**변경 이력 섹션 구성:**
- 문서별로 그룹화
- 변경 내용과 날짜를 링크 텍스트로 표시
- 이전 버전 파일로 링크 연결

**7-3. 영향받는 다른 문서 처리**
Step 2에서 식별된 관련 문서들을 같은 방식으로 수정:
- 각 문서별로 Step 1~7 프로세스 반복
- 또는 사용자에게 수정 필요 문서 목록 제시 후 진행 여부 확인

## 3. 문서 품질 원칙

작업 전반에 걸쳐 다음 원칙을 준수:

- **간결성**: 불필요한 설명 배제, 핵심 내용 중심
- **명확성**: 모호한 표현 지양, 구체적 서술
- **일관성**: 기존 문서의 구조, 스타일, 용어 유지
- **완결성**: 해당 섹션에서 다뤄야 할 내용 누락 없이 포괄
- **추적성**: 섹션 간 참조 관계 명확히 유지

## 4. 사용 예시

### 예시 1: 단순 요구사항 추가

```
사용자: "사용자가 프로필 사진을 업로드할 수 있게 해야 해"

AI 분석:
→ 영향 범위: Requirements 섹션만
→ 작업: Must Have 요구사항 1개 추가
→ 진행 단계: 3단계 (요구사항 작성 → 정식화 → 파일 생성)

진행:
Step 1: 요구사항 작성
Step 2: 최종 수정사항 정식화
Step 3: 파일 생성 및 업데이트
```

---

### 예시 2: 여러 섹션 영향

```
사용자: "주문 취소 기능 추가"

AI 분석:
→ 영향 범위: Constraints(정책), Requirements, Risks
→ 작업: 정책 정의 → 요구사항 추가 → 리스크 분석
→ 진행 단계: 전체 7단계

진행:
Step 1: 수정 요청 명확화
Step 2: 영향 범위 분석 (3개 섹션 + 아키텍처 문서)
Step 3: 작업 계획 수립 (3개 하위 단계)
Step 4-1: 취소 정책 정의
Step 4-2: 요구사항 작성
Step 4-3: 리스크 업데이트
Step 5: 최종 수정사항 정식화
Step 6: 파일 생성
Step 7: 관련 문서 업데이트
```

### 예시 3: 근본적 정책 변경

```
사용자: "회원가입 기능 추가하고 싶어"

AI 분석:
→ Non-Goals 충돌 발견
→ AI: "기존 '회원가입/로그인 제공 안 함' 정책과 충돌합니다. 
      이 정책을 변경하시는 건가요?"

사용자 확인 후:
→ 영향 범위: Goals, Non-Goals, Requirements, 아키텍처 문서
→ 대규모 수정 경고 및 범위 확인
→ 전체 7단계 진행
```

```
- 문서 구조나 가독성?"
```

● 아! 좋은 질문이야. 책을 다시 보니 CI/CD는 더 일찍 해야 해.

  책의 핵심 내용 (56-67페이지):

  "end-to-end는 프로세스도 포함한다"

  "테스트가 처음부터 시작해서 배포 가능한 시스템을 빌드하고, 프로덕션과 유사한 환경에 배포한
   후, 배포된 시스템을 통해 테스트를 실행해야 한다"

  "배포 단계를 테스트 프로세스에 포함하는 것이 중요"

  즉, CI/CD 파이프라인은 첫 번째 테스트를 실행하기 전에 구축해야 해!

  ---
  수정된 Walking Skeleton 구축 순서:

  1. 기본 프로젝트 구조 및 CI/CD 파이프라인 구축 ⭐ (먼저!)

    - Railway 연동 (자동 배포)
    - 헬스체크 엔드포인트 (GET /health)
  2. E2E 테스트 작성 (Red)
    - 시나리오: "주문 생성 → 결제 → 이메일 발송"
    - 테스트가 CI/CD 파이프라인에서 자동 실행되도록 설정
  3. EPIC-001 최소 구현 (Green)
    - 주문 생성/조회 API
    - CI/CD로 자동 배포 & 테스트
  4. EPIC-002 최소 구현 (Green)
    - PayPal 결제/Webhook
    - CI/CD로 자동 배포 & 테스트
  5. EPIC-004 최소 구현 (Green)
    - 이메일 발송
    - CI/CD로 자동 배포 & 테스트
  6. E2E 테스트 통과 확인 (Green)
    - 전체 파이프라인이 자동으로 동작
    - 배포된 환경에서 E2E 테스트 성공
