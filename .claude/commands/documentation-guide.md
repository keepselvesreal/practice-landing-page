## 문서화 원칙

- 독립적 이해 가능성: 이 문서만 봐도 문제~해결 전체 흐름 파악 가능
- 구체성: 파일명:라인 번호 명시, 코드 변경 전후(Before/After) 표시
- 쉽고 명확한 용어 설명: 기술 용어를 모르는 사람도 이해 가능하게
- 선택적 작성: 해당하는 내용이 있을 때만 작성 (기계적으로 모든 섹션 채우지 않음)

## 문서 구조

### 작업 내용 (필수)

#### 작업 맥락
- 배경, 요구사항, 목표
- 왜 이 작업을 하게 되었는가?

#### 작업 분석
- 설계, 접근 방식
- 기술 선택 이유
- 현재 상황 맥락 포함

#### 수행 내용
- 구현 세부사항
- **파일**: 변경한 파일 경로
- 코드 변경 Before/After
- 핵심 한 줄 설명

---

### 이해 요소 (선택)

기술적 개념을 **모르는 사람도 이해 가능하도록** 설명

**템플릿**:
```markdown
### [개념/원리 제목]

**핵심**: [한 문장 요약]

**동작 원리**:
- [단계별 설명]

**코드 예시**: (필요시)
```

---

### 판단 요소 (선택)

**템플릿**:
```markdown
### [판단한 주제]

**선택지**:
- A 방법: [설명]
- B 방법: [설명]

**선택**: [선택한 방법]

**이유**:
- [현재 상황 맥락 포함]
- [트레이드오프 고려]

**결과**: [실제 적용 후 결과]
```

---

### 개선 요소 (선택)

현재 한계를 명확히 서술

**템플릿**:
```markdown
#### 함수/클래스 차원
**현재 한계**: [구체적 문제점]
**개선 방향**: [해결 방안]

#### 아키텍처 차원
**현재 한계**: [구조적 문제점]
**개선 방향**: [리팩터링 방향]
```

---

### 예외 요소 (선택)

예상하지 못한 문제나 오류가 있었을 경우에만 작성

**템플릿**:
```markdown
### [오류/문제 제목]

**증상**: [구체적 현상, 에러 메시지]

**원인**: [왜 발생했는가]

**해결**: [어떻게 해결했는가]

**교훈**: [다음에 주의할 점]
```
