# Chapter 1: Prologue - 추출된 정보

## 핵심 내용
- ATDD를 통해 요구사항을 테스트 가능한 형태로 정제하는 과정을 소개
- 개발자·테스터·고객으로 구성된 트라이어드가 어떻게 협업하는지 대비 사례로 설명
- 초기 테스트 부재가 반복 작업과 결함 보고를 초래하는 방식과 대비해 선제적 테스트의 이점 강조
- 테스트 가능한 요구 정의가 리뷰와 인수 과정을 매끄럽게 만드는 흐름 제시

## 상세 핵심 내용

### 두 가지 개발 플로우 비교
- 전통적 방식은 요구 이해 부족과 의사소통 단절로 인해 반복적인 결함 수정이 발생
- ATDD 방식은 구현 이전에 합의된 시나리오 테스트를 만들고 이를 통해 완료 기준을 명확히 함

### 트라이어드의 역할
- 고객(Cathy)은 요구를 명확히 하고 테스트 기대 결과를 공유
- 개발자(Debbie)는 테스트를 통과하도록 구현하며 테스트 자동화 기반을 마련
- 테스터(Tom)는 테스트 설계 관점에서 위험 영역을 찾아내고 시나리오의 공백을 채움

### 테스트 가능한 요구의 가치
- 미리 정의된 인수 기준은 리뷰 시간을 줄이고 재작업 비용을 감소
- 공통 언어로 작성된 테스트는 의사소통 도구이자 품질 게이트 역할을 수행

## 상세 내용

### 초기 결함 루프와 재작업 비용
- 구현 후 발견되는 결함은 다음 반복의 시간을 잠식하고 팀 사기를 저하시킴
- “마지막 날” 테스트는 요구 해석의 격차를 은폐한 채 진행되어 위험이 늦게 드러남

### 선행 테스트와 흐름의 안정성
- 개발자는 구현 중 수시로 인수 테스트를 실행하며, 완료 전 검증을 수행
- 테스트 자동화는 반복 실행을 가능하게 하여 회귀 위험을 낮춤

### 요구 명세에서 테스트로의 전환
- 고객과 테스터는 요구를 사용자 시나리오로 전환하여 개발자가 바로 사용할 수 있도록 만듦
- 시나리오 형태는 비즈니스 언어를 유지하면서 “완료”의 정의를 구체화함

## 주요 화제

### ATDD 도입 동기
- 반복적인 결함 보고와 의사소통 단절이 비즈니스 가치를 지연시킴
- 테스트를 요구 정의의 일부로 통합하면 품질과 속도를 동시에 확보할 수 있음

### 협업 리듬과 리뷰
- 구현 전·후에 트라이어드가 만남으로써 피드백 루프를 짧게 유지
- 리뷰는 테스트 결과를 중심으로 진행되므로 주관적 논쟁이 줄어듦

### 공감 기반의 언어 정렬
- 동일한 단어도 각 역할마다 다른 의미를 가질 수 있음을 사례로 설명
- 테스트 사례를 통해 용어 차이를 조정하고 기대치를 맞춤

## 부차 화제

### 자동화 수준과 실행 주기
- 수동 테스트에서 자동화된 스크립트로 이동하며 반복 실행 가능성 확보
- 빌드 파이프라인에 인수 테스트를 포함시킬 필요성 시사

### 역할 간 학습과 상호 보완성
- 개발자도 테스트 관점을 익히고 테스터도 구현 세부를 이해해야 한다는 메시지 전달
- 요구 명세를 시나리오화하면서 고객도 시스템 행동을 더 깊이 이해하게 됨
