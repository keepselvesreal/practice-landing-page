# Chapter 4: An Introductory Acceptance Test - 추출된 정보

## 핵심 내용
- 실제 할인 규칙 사례를 통해 요구를 테스트 가능한 예제로 전환하는 과정을 설명
- 모호한 비즈니스 규칙을 예제 테이블로 명확화하고 이해관계자 승인을 받는 절차를 제시
- 인수 테스트를 실행하는 네 가지 접근(수동 스크립트, 테스트 UI, 단위 테스트 프레임워크, 전용 AT 프레임워크)을 비교
- 예외 조건·경계값을 빠르게 발견하기 위한 질문과 테스트 설계 기법을 보여줌

## 상세 핵심 내용

### 예제 기반 명세화
- 규칙 문장을 그대로 두지 않고 입력/출력 테이블로 재구성해 해석 차이를 줄임
- 경계값($10.00, $50.00)과 고객 등급에 따른 기대값을 명시하여 구현 가이드 제공

### 합의와 확인 절차
- ??와 같은 표식을 사용해 불확실 항목을 표시하고 이해관계자에게 재확인 요청
- 음수 금액과 같이 예상 밖 케이스를 테스트에 포함시켜 정책 결정을 촉진

### 실행 옵션 비교
- 수동 스크립트: 실제 UI 흐름을 그대로 재현하지만 반복 실행 비용이 큼
- 테스트 전용 UI: 계산 로직만 빠르게 검증 가능, 사용자 경험은 별도 확인 필요
- 단위 테스트 프레임워크: 개발자가 코드 수준에서 예제를 자동화
- AT 전용 프레임워크: 비즈니스 언어로 테스트를 표현하고 협업 가독성을 강화

## 상세 내용

### 시나리오 테이블 설계
- 각 열은 입력·상태, 최종 열은 기대 결과로 구성하여 문장보다 빠른 해석 가능
- 좋은/우수 고객과 주문 금액 조합을 완전탐색하여 요구 공백을 채움

### 예외 처리 정책 결정
- 음수 금액, 미정의 고객 등급 등 비정상 시나리오를 테스트 제안으로 제시
- 정책 결정 후 테스트가 업데이트되며 요구 베이스라인이 갱신됨

### 테스트 실행 도구 선택 가이드
- 선택 기준: 반복성, 유지보수성, 사용자 환경 모사 여부
- 자동화 수준이 높을수록 빠른 회귀가 가능하나, UI·경험은 별도 검증이 필요

## 주요 화제

### 예제 기반 사양(Example Mapping)
- 비즈니스 규칙을 작은 예제로 분해해 이해를 공유하는 방법을 시연

### 경계값 분석
- “less than or equal to” 표현의 해석 차이를 테이블로 검증하고 경계 테스트 중요성 강조

### 테스트 설계 협업
- 테스터가 반례를 제시하고 고객이 정책을 명확히 하며 개발자가 구현 가능성을 검토하는 상호작용 묘사

## 부차 화제

### 테스트 자동화 수준 로드맵
- 초기에는 수동으로 시작해 반복 시 전용 도구나 프레임워크로 옮겨가는 전략 제안

### 문서화와 트레이서빌리티
- 인수 테스트가 요구 사항 문서와 회귀 스위트 역할을 동시에 수행함을 언급
