# Chapter 32: Case Study – E-Mail Addresses - 추출된 정보

## 핵심 내용
- 복잡한 이메일 검증 규칙을 작은 테스트 세트로 분할해 구현/이해 시간을 단축하는 방법 제시
- RFC 2822 규칙을 기반으로 로컬 파트·도메인·금지 도메인 규칙 등을 따로 정의
- 각 규칙을 표와 예제 케이스로 표현해 모듈화된 테스트(및 코드) 설계를 지원

## 상세 핵심 내용

### 원래 테스트 세트
- 다양한 유효/무효 이메일을 단일 표로 관리; 유지보수 비용과 구현 난이도가 높았음
- 일부 도메인 금지 등 비즈니스 규칙도 포함

### 규칙 분해 전략
1. `@` 포함 개수 확인(1개만 허용) → 로컬/도메인 분리
2. 로컬 파트 허용 문자/길이 규칙 정의(특수 문자, 점 위치 등)
3. 도메인 형식 규칙, TLD 요구 사항 등 별도 표로 정리
4. 금지 도메인 목록 테스트로 정책 규정
- 각 단계는 짧은 테스트로 구현 가능, 누락 시 빠르게 파악

### 테스트 설계 이점
- 개발자가 작은 단위부터 구현하며 빠른 피드백을 얻고, 복잡 로직의 회귀 범위를 줄임
- 테스트 이름과 표 구조가 요구 문서 역할을 동시에 수행

## 주요 화제

### 규칙 기반 접근
- 표 형태 규칙은 실무자·테스터에게 요구 정의와 구현 가이드를 동시에 제공

### 보안/정책 확장
- SQL 인젝션 등 보안 고려로 특정 문자 제외, 금지 도메인 정책 등 추가 규칙 반영 가능

## 부차 화제

### 이메일 플러스 주소 트릭
- `name+tag@domain` 패턴 허용 여부를 설명하며 테스트/정책 설정 시 주의사항 언급
