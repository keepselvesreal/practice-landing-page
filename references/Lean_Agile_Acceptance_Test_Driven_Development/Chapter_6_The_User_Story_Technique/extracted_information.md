# Chapter 6: The User Story Technique - 추출된 정보

## 핵심 내용
- 기능을 보다 작은 사용자 스토리로 분할해 테스트 가능한 단위로 만드는 과정 설명
- 역할·속성·페르소나 정의를 통해 스토리 작성 시 관점 다양성을 확보
- 스토리 카드 템플릿(As a … I want … so that …)과 수용 기준 정의법 소개
- 스토리 크기 결정과 분할에 acceptance criteria, INVEST 원칙, 가치/노력 추정 활용

## 상세 핵심 내용

### 역할과 페르소나 도출
- 브레인스토밍으로 역할 목록을 작성하고 책임을 명확히 함
- 역할 속성(사용 빈도, 도메인/컴퓨터 전문성, 일반 목표)을 정의하여 설계·테스트 요구 파악
- 세부 페르소나를 만들어 사용자 경험을 구체적으로 상상하고 UI/테스트 기준을 조정

### 스토리 작성과 수용 기준
- XP 스타일 스토리 문법을 사용하여 행위자·행동·목적을 간결히 표현
- 스토리마다 초기 인수 기준을 작성하여 구현 범위와 테스트 초점을 구체화
- 인수 기준이 복잡하거나 다수일 경우 스토리를 더 작은 단위로 재분할

### INVEST와 추정
- 독립성(Independent): 각 스토리가 다른 스토리에 과도하게 의존하지 않도록 설계
- 협상 가능성(Negotiable)과 가치(Valuable): 고객과 협의하여 우선순위 판단, 비즈니스 가치 추정
- 추정 가능성(Estimable), 소형(Small), 검증 가능(Testable): 노력 산정과 테스트 가능성을 기반으로 현실적인 사이즈 결정

## 상세 내용

### 스토리 분해 전략
- 큰 기능(예: Charge Rental)을 “Compute Rental Fee”, “Process Card Charge” 같은 하위 스토리로 분해
- 비즈니스 스토리에 연결되지 않는 기술 스토리는 가치를 재검토

### 브레인스토밍과 카드 정렬
- 아이디어를 카드로 적어 그룹화함으로써 중복 제거와 범주화 수행
- 스토리 벽 또는 도구에 간단한 설명만 기록하고 세부사항은 구현 직전에 수집

### 가치와 노력 추적
- 스토리별 비즈니스 가치와 노력 추정치를 기록하여 ROI 기반 우선순위 조정 가능
- 누적 가치 추적을 통해 진행 상황을 시각화하고 성공 지표로 활용

## 주요 화제

### 역할 기반 테스트 설계
- 각 역할/페르소나에 맞는 테스트 시나리오를 준비하여 UX 요구를 반영

### 스토리 수용 기준과 테스트의 상호작용
- 인수 기준 수가 스토리 규모 판단의 신호가 될 수 있음을 강조

### 협업 리듬 유지
- 정기적인 스토리 분해 세션으로 백로그를 필요한 만큼만 준비하는 Just-In-Time 접근 소개

## 부차 화제

### 악의적 역할 스토리
- 악의 사용자 시나리오를 통해 보안/오용 테스트를 고려하자는 제안

### 개발자 스토리와 비즈니스 가치
- 기술적 스토리도 비즈니스 스토리에 연결되어야 하며, 가치 추적 근거를 남겨야 함을 강조
