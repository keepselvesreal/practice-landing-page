# Chapter 12: Development Review - 추출된 정보

## 핵심 내용
- 스폰서와의 개발 리뷰 미팅을 통해 현재 구현, 테스트, 남은 작업을 공유
- 데이터 관리(CRUD) 스토리, 비즈니스 규칙(중복 방지, 비활성화 등)의 중요성을 사례로 설명
- 사용자 피드백을 반영한 사용성 개선과 상태/표시 분리 전략을 소개
- 품질 속성 테스트(성능, 보안 등)와 출력 검증 템플릿을 통해 전체 품질 확보

## 상세 핵심 내용

### 개발 진행 상황 공유
- 주요 스토리(체크아웃, 체크인, 카드 청구) 외에도 고객/상품 관리 기능을 추가 구현
- 데이터 삭제는 비활성화로 처리해 이력 유지 및 참조 무결성 확보
- 유스케이스별 인수 테스트를 UI와 미들티어에서 실행해 정합성 확인

### 사용성 및 접근성 개선
- 실제 사용자(색각 장애, 시력 등) 피드백을 바탕으로 메시지 색상, 폰트 크기, 텍스트 명확화 조정
- 계약서 문구 검토를 통해 이해 가능한 언어로 전환, 승인 전까지는 데이터만 검증

### 상태와 표시 분리
- 시스템 내부 상태는 식별자 기반(CD_Rental_Limit_Exceeded)으로 테스트하고, 표시 텍스트는 별도 맵핑
- 같은 로직을 다양한 입력 장치(키보드, 바코드, RFID)와 독립적으로 유지하도록 설계

## 상세 내용

### 출력 검증 구조화
- 계약서 출력 테스트를 템플릿+데이터 조합으로 표현해 변경 시 영향 범위 최소화
- 메시지 매핑 테이블을 통해 문구 승인 전에도 기능 테스트 가능

### 품질 속성 테스트 준비
- 동시 체크아웃 수 대비 허용 응답 시간을 표로 정의해 성능 목표 설정
- 보안·물리적 접근 통제 등 비기능 요구에 대한 점검 필요성 강조

### 리뷰 미팅 운영
- 고객과 테스터가 공동으로 테스트 진행 상황을 설명하고, 스폰서가 남은 우선순위를 조정
- 완료된 테스트 목록과 탐색 테스트 결과를 투명하게 공유해 신뢰 구축

## 주요 화제

### CRUD 스토리의 암묵적 필요
- 핵심 기능 외에도 테스트/운영에 필수인 데이터 관리 스토리가 자동으로 파생됨을 강조

### 사용자 관점 테스트
- 다양한 사용자 페르소나를 직접 참여시켜 UI/출력품의 이해도를 검증하는 실무 팁 제공

### 테스트 유지보수 전략
- 문구 등 빈번히 바뀌는 요소를 데이터화해 코드 변경 없이 테스트만 업데이트하도록 분리

## 부차 화제

### 법무/규제 협업
- 변호사 검토가 필요한 산출물은 임시 텍스트로 테스트하고 승인 후 갱신하는 절차 제안

### 성능·보안 가정 검증
- 향후 사용자 수 증가에 대비해 부하 목표를 정의하고, 물리적 보안 위협도 고려해야 함을 언급
