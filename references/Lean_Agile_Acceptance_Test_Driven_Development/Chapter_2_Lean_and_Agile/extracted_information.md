# Chapter 2: Lean and Agile - 추출된 정보

## 핵심 내용
- 트라이어드(고객·개발·테스트) 유닛 구조와 상호 책임을 소개
- 전통적 사후 검증 흐름과 ATDD 선행 테스트 흐름을 비교하며 피드백 루프 차이를 설명
- ATDD가 린의 낭비 제거 및 학습 증폭, 애자일의 협업과 빠른 피드백 원칙과 어떻게 맞닿는지 분석
- 고객 응답 지연이 설계와 납기 위험을 초래함을 경고하고 신속한 커뮤니케이션 규약을 제시

## 상세 핵심 내용

### 유닛 기반 협업 모델
- 고객 유닛은 요구 정의와 인수 기준 설정, 우선순위 결정을 책임짐
- 개발 유닛은 요구를 구현하고 테스트 자동화를 통해 품질을 보장
- 테스트 유닛은 위험 탐지와 인수 테스트 보강으로 검증 깊이를 확보

### 사후 테스트 vs 선행 테스트
- 사후 테스트 흐름은 요구 오해로 인한 반복 작업과 비용 증가를 초래
- 선행 테스트 흐름은 합의된 시나리오를 바탕으로 구현하며 피드백 사이클을 단축

### Lean·Agile 원칙과 ATDD
- 낭비 제거: 구현 이전에 테스트를 정의해 반복 수정 루프를 줄임
- 내재적 품질: 모듈 수준에서 인수 테스트를 통합해 일관된 품질 확보
- 학습 증폭: 트라이어드 협업을 통해 도메인 지식과 기술 이해가 교차 학습됨
- 애자일 가치: “비즈니스와 개발의 일상적 협업”과 “작동하는 소프트웨어” 지표를 충족

## 상세 내용

### 피드백 속도 관리
- 빠른 질문·응답 채널이 설계 결정 지연을 방지하고 다른 업무로의 전환 비용을 최소화
- 필수 정보와 지연 허용 정보의 구분을 통해 고객과 팀 모두의 시간을 보호

### 커뮤니케이션 언어 조정
- 동일 용어라도 역할별 해석 차이가 존재하므로 예시와 테스트 데이터를 통해 의미를 고정
- 시각 자료(플로우 차트, 피규어)를 활용해 기대되는 프로세스를 공유

### 품질 측정과 보고
- 합의된 테스트 세트가 진행 상황을 가시화하여 “완료”를 명확히 정의
- 통과/실패 결과는 고객 리뷰와 배포 의사결정의 근거가 됨

## 주요 화제

### 트라이어드의 조직적 위치
- 프로젝트 팀을 공격·수비 유닛 비유로 설명하며 역량 중첩과 역할 유연성 강조

### Lean 원칙의 실무 적용
- 낭비, 흐름, 지속가능한 페이스 등을 사례와 함께 소개하여 ATDD 실행 맥락 제공

### 애자일 가치의 재해석
- 애자일 선언문 원칙을 ATDD 관점에서 재해석하여 팀이 가치와 연결될 수 있도록 함

## 부차 화제

### 고객 응답 SLA 제안
- 질문의 긴급도를 분류하고 응답 지연이 일정에 미치는 영향을 수치화해 설명

### 기타 이해관계자 소개 예고
- 챕터 후반에서 차후에 등장할 이해관계자(스폰서, 사용자 등)의 역할을 예고하여 맥락 확장
