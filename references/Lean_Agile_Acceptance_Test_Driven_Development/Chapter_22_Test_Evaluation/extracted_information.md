# Chapter 22: Test Evaluation - 추출된 정보

## 핵심 내용
- 좋은 테스트의 특성: 고객 친화성, 안정성(아이도폰트, 비취약성), 단일 개념 검증 등
- 테스트 구성 요소(설정, 트리거, 확인)가 명확하며, 최소 기대 결과만 확인해 유지보수성을 높임
- 환경·외부 의존성 제어, 우선순위별 분리, 오케스트레이션을 통해 반복 가능성 확보

## 상세 핵심 내용

### 고객 이해 가능성
- 도메인 언어 사용, 오타 제거, 고객이 선호하는 표현(표/그래픽 등) 채택
- 단위 테스트 프레임워크로 구현하더라도 기대 행동이 고객 표현과 매핑되도록 주석·명명

### 안정적 테스트
- 테스트는 항상 동일 결과를 내야 하며, 상태 공유·시간 의존성·랜덤 요소는 테스트 더블로 제어
- 공통 셋업 변경에 민감한 테스트는 사전 조건 검증을 추가해 실패 원인 파악 용이화

### 최소 검증·관심사 분리
- 필요한 결과만 확인하여 부수적 변경으로 인한 실패를 줄임
- 비즈니스 규칙·표시·워크플로 단계 등 관심사를 나누어 별도 테스트 유지

### 환경 검증
- 필요한 플랫폼 버전·구성 등을 테스트 전 확인해 실행 오류 대신 명확한 실패 메시지 제공

## 주요 화제

### 테스트 설계 가이드
- 메사로스, 커닝햄 등의 원칙을 바탕으로 테스트 품질 체크리스트 제공

### 테스트 조건 관리
- 사전조건·작업·결과를 명시해 실패 원인(설정 vs 동작 vs 검증) 구분을 쉽게 함

## 부차 화제

### 모듈 단위 유효성 테스트
- 공통 기능(예: 고객 ID 포맷)을 독립 테스트하여 다른 테스트가 중복 검증하지 않도록 함

### 테스트 문서화 품질
- 맞춤법 검사, 약어 사전 등 커뮤니케이션 품질 관리도 테스트 자산의 일부로 강조
