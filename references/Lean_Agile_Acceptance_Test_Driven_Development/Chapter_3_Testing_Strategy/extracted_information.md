# Chapter 3: Testing Strategy - 추출된 정보

## 핵심 내용
- 전체 테스트 전략 속에서 ATDD가 차지하는 위치를 Gerard Meszaros의 테스트 매트릭스로 설명
- 고객 테스트, 속성(비기능) 테스트, 사용성·탐색적 테스트, 개발자 테스트의 목적과 실행 방식을 정리
- 테스트 실행 환경(개발자 PC, 통합, 테스트, 운영)별로 어떤 테스트가 돌아가는지와 위험 관리 지표를 제시
- 긍정·부정 테스트, 자동화도, 테스트 커버리지의 균형이 품질 확보에 어떻게 기여하는지 논의

## 상세 핵심 내용

### 테스트 분류와 목적
- 고객 테스트는 비즈니스 관점을 검증하며 대부분 예/아니오로 판별 가능한 시나리오
- 속성 테스트는 성능·보안 등 비기능 요구를 다루며 일부는 정량화, 일부는 추정에 의존
- 사용성 테스트는 정성 평가가 중심이며 자동화가 어려움
- 탐색적 테스트는 사전 스크립트 없이 병행 설계·실행·분석을 반복하며 새로운 결함을 발견
- 유닛·컴포넌트 테스트는 설계 품질과 코드 이해를 높이고 ATDD 시나리오로부터 파생될 수 있음

### 테스트 실행 플랫폼
- 개발자 머신: 단위 및 일부 인수 테스트를 빠르게 실행, 외부 의존성은 대체물(테스트 더블) 활용
- 빌드/통합 서버: 병합 후 회귀를 검증하고 공유 인수 테스트를 자동화
- 테스트 환경: 전체 외부 시스템과 함께 시나리오, 탐색적, 속성 테스트 수행
- 운영 환경: 탈출 버그(escaped defects) 수를 품질 지표로 추적

### 테스트 파셋과 균형
- 긍정 테스트는 요구 충족 여부를 확인하고, 부정 테스트는 오류 처리 경로를 검증
- 자동화 가능한 부분과 수동 검증이 필요한 부분을 구분해 효율과 신뢰성을 동시에 확보
- 테스트 커버리지, 결함 발견 시점, 비즈니스 가치 전달 속도의 상관관계 분석

## 상세 내용

### 테스트 매트릭스 활용
- 비즈니스/기술 축과 개발 지원/제품 비판 축으로 테스트 목적을 시각화
- ATDD는 비즈니스-제품 비판 영역에서 반복 실행 가능한 실행 명세로 기능함

### 테스트 데이터와 판정 기준
- 예/아니오 판단 기준을 통해 가독성과 자동화 가능성을 높이고, 모호한 질적 요구는 별도 다룸
- 비기능 요구는 목표 값과 허용 한계(예: 응답 시간 < 5초)를 명시해 측정 가능성을 확보

### 탐색적 테스트의 트라이어드 참여
- 고객·개발자·테스터가 사용자의 페르소나를 맡아 시나리오를 걷는 방식으로 숨겨진 문제 탐색
- 시스템이 동작 가능한 상태에서 언제든 수행 가능하며, 테스트 노트로 학습을 축적

## 주요 화제

### 테스트 자동화 전략
- xUnit, 특수 목적 도구 등 자동화 스택을 소개하고 테스트 유형별 적합성을 점검

### 테스트 거버넌스
- 각 플랫폼에서 어떤 테스트를 필수로 실행해야 하는지와 책임 소유자 지정

### 품질 지표의 추적
- 탈출 버그 수, 테스트 성공률, 실행 빈도 등 지표를 통해 품질 트렌드를 모니터링

## 부차 화제

### 테스트 더블의 도입 예고
- 외부 의존성 격리를 위해 추후 챕터에서 다룰 테스트 더블 개념을 미리 언급

### 사용자 연구 리소스 연계
- 사용성 테스트에 UX 전문가, 고객 대표 등이 참여해야 함을 언급하며 범위를 넓힘
