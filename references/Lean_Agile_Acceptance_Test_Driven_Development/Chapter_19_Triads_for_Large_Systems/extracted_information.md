# Chapter 19: Triads for Large Systems - 추출된 정보

## 핵심 내용
- 대규모 시스템에서는 다양한 역할 조합으로 트라이어드를 확장하여 영역별 전문성을 반영
- 아키텍트·데이터베이스 전문가 등 기술 역할이 참여한 트라이어드는 모듈 책임과 기술 요구를 검증
- 일부 기술 프로젝트(버전 업그레이드 등)는 고객 테스트 없이도 기존 인수 테스트를 회귀로 활용
- 기존 시스템 인수·벤더 제품 적용 시 테스트 부재 문제에 대응하는 전략을 제시

## 상세 핵심 내용

### 트라이어드 확장
- 도메인 전문가-개발자-테스터 조합 외에 아키텍트 포함 기술 트라이어드, DB 아키텍트 포함 트라이어드 등 소개
- 각 트라이어드는 해당 영역(아키텍처, 데이터 지속성 등)의 수용 기준을 정의·검증

### 아키텍처 검증
- 초기 단계에 엔드-투-엔드 고객 테스트를 동작시켜 아키텍처 적합성을 확인하고 회귀 기준 마련
- 모듈화 결정은 고객 인수 테스트를 분해해 책임을 배분하며 진행

### 기술 스토리와 테스트
- 데이터베이스 전환·플랫폼 업그레이드 등 고객 기능 변화가 없는 작업은 기존 인수 테스트를 회귀로 사용
- 기술 리더가 “내부 고객” 역할을 맡아 품질 기준과 완료 조건을 정의해야 함

### 테스트 부재 대응
- 벤더 제품: 구성/커스터마이징 범위를 이해하고 필요한 경우 자체 인수 테스트를 설계
- 인수 시스템: 기존 동작을 기록·자동화해 회귀 기준을 만든 후 변경 진행

## 주요 화제

### 역할 전문화
- 대규모 팀에서 역할이 세분화되므로 의사소통 채널과 책임 명확화가 필수

### 회귀 테스트의 가치
- 고객 테스트 유무와 관계없이 자동화 회귀 스위트는 기술 변경의 안전망 역할 수행

### 벤더 협업 전략
- 계약 시 테스트 증거 제공 요구, 구성 변경 시 리스크 기반 테스트 설계 필요

## 부차 화제

### 시스템 인수 시 초기 평가 체크리스트
- 테스트 범위, 데이터 품질, 구성 가능성 등을 빠르게 진단해 리스크를 파악

### 기술 프로젝트 분리 권장
- 고객 기능과 무관한 대규모 기술 작업은 별도 프로젝트로 분리해 우선순위·리스크 관리
