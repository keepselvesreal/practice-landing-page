# Chapter 18: Entities and Relationships - 추출된 정보

## 핵심 내용
- 비즈니스가 사용하는 실체(앨범, CD, 대여 등)를 명확히 구분하고 관계를 모델링해 요구를 정제
- UML 등 모델 다이어그램과 표를 연계해 다대일, 다대다 관계를 표현하고 테스트 데이터 구조화
- 엔터티 분리(예: Rental 엔터티 분리)를 통해 중복을 줄이고 테스트 설정을 단순화

## 상세 핵심 내용

### 관계 모델링
- 앨범과 CD의 관계(한 앨범에 여러 CD, 각 CD는 한 앨범)의 다이어그램과 표 매핑
- UPC 코드와 같은 키를 관계 식별자로 사용하여 데이터 일관성을 확보

### 엔터티 분해의 이점
- 기존 CD 테이블에 포함된 `Rented` 상태를 별도 `Rental` 엔터티로 이동해 책임을 분리
- Rental 테이블이 고객·CD를 연결하며 체크인/체크아웃 테스트 셋업을 간소화

### 표 기반 테스트 표현
- 모델 다이어그램의 관계를 표로 치환해 테스트 케이스, 셋업 데이터를 명확히 표현
- 빈 필드를 허용하기보다 레코드 존재 자체로 상태를 나타내어 데이터 해석을 명료화

## 주요 화제

### 비즈니스 용어 정렬
- 모델은 구현 관점이 아닌 비즈니스에서 사용하는 개념/이름을 따르며, 테스트에서도 동일 용어 사용

### 정규화와 테스트 효율
- 중복 제거와 엔터티 분리를 통해 변경 영향 범위를 줄이고 테스트 유지보수성을 높임

### 다이어그램-테이블 변환
- UML 등 시각 표현을 테스트 데이터로 빠르게 변환하는 절차를 소개

## 부차 화제

### 추가 엔터티 도입 가능성
- 예약, 판매 등 확장 기능 도입 시 새로운 엔터티·관계 추가 필요성을 암시

### 설계 승인 프로세스
- 모델 변경은 트라이어드 합의가 필요하며, 승인 후 테스트/데이터 구조를 업데이트
