# Chapter 30: How Does What You Do Fit with ATDD? - 추출된 정보

## 핵심 내용
- 개발/테스트 실무 관점에서 ATDD 적용 시 테스트 계층 구성, 설계 패턴, 디바이스 검증 등을 정리
- 테스트 실행 주기를 플랫폼별로 조정하고 스모크→장기 테스트까지 체계화하는 전략 제시
- 테스트가 내부 설계(모델-뷰 분리 등)를 유도하며, 외부 장치 검증은 개발자 인수 테스트로 보완

## 상세 핵심 내용

### 테스트 플랫폼/주기 관리
- 개발자 로컬의 빠른 테스트, 15분 빌드 스위트, 야간/주말/주간 장기 테스트 등 단계별 파이프라인 구성
- 긴 러닝 타임을 가진 시스템(컴파일러 등)을 위한 장기 테스트 필요성 강조

### 설계와 테스트의 상호작용
- 모델-뷰-컨트롤러 분리를 통해 비즈니스 로직 테스트와 UI 테스트를 독립시키고 단순화
- `allowedToReserve` 등 명확한 메서드/이름을 통해 테스트와 코드의 가독성 향상

### 디바이스/외부 의존성 테스트
- 바코드 스캐너 등 하드웨어는 개발자 인수 테스트(수동 혹은 자동화)로 기능 범위를 검증
- 고객 인수 테스트는 통합 여부 확인 정도로 최소화해 중복을 줄임

## 주요 화제

### 테스트 우선 설계
- 간단한 테스트가 단순한 코드 구조로 이어지며, 경계 조건은 개발자 테스트로 보완

### 장치·입력 채널 확장
- 새로운 입력 장치 도입 시 고객 테스트는 변경 최소화, 개발자가 장치 특화 검증 수행

## 부차 화제

### 실패 사례 교훈
- 항공사 예약 시스템 마이그레이션 실패 사례를 통해 충분한 테스트의 중요성 환기

### 사용자 인터페이스 테스트 전략
- UI부터 시작할 경우에도 핵심 비즈니스 규칙 테스트를 별도로 유지해 설계 유연성 확보
